# Feature Specification: Timezone Scheduler CLI

**Feature Branch**: `001-timezone-scheduler-cli`

**Created**: 2026-07-02

**Status**: Draft

**Input**: User description: "建立一個命令列工具（Command Utility），讓使用者可以透過時區、地點名稱或郵遞區號（如適用於美國）等任何能提升易用性的方式，來查詢指定地點的目前日期與時間。此外，請加入所有有助於輕鬆進行跨時區會議排程的相關功能。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 查詢目前日期與時間 (Priority: P1)

使用者能輸入時區、地點名稱或美國郵遞區號，快速查詢該地點目前的日期、時間與相對 UTC 的偏移量。

**Why this priority**: 這是工具的基本核心功能，為後續時間轉換與會議排程提供基礎定位。

**Independent Test**: 執行 `tz-cli query --location "Taipei"`，輸出應顯示台北的目前日期、時間與時區（例如 UTC+08:00）。

**Acceptance Scenarios**:

1. **Given** 系統正常運作，**When** 使用者執行 `tz-cli query --location "Tokyo"`，**Then** 系統應精確輸出東京當前的西元日期與 24 小時制時間，並顯示時區為 `Asia/Tokyo`。
2. **Given** 系統正常運作，**When** 使用者輸入美國郵遞區號 `94105`，**Then** 系統應辨識出該郵遞區號屬於 San Francisco 區域，並正確輸出太平洋時間（PT）的日期與時間。
3. **Given** 系統正常運作，**When** 使用者輸入時區標識符 `Europe/London`，**Then** 系統應顯示倫敦時間，並正確考量當前是否處於夏令時間（BST）。

---

### User Story 2 - 跨時區時間轉換與對照 (Priority: P2)

使用者可以輸入一個特定的時間點與來源地點，並指定多個目標地點，工具會輸出該時間在各目標地點的對應時間對照表。

**Why this priority**: 協助跨國團隊成員確認特定會議時間在各自時區代表的具體時間，避免換算錯誤。

**Independent Test**: 執行 `tz-cli convert --time "2026-07-03 14:00" --from "Taipei" --to "London" --to "New York"`，應在終端機輸出包含倫敦（當天 07:00）與紐約（當天 02:00）的對照表。

**Acceptance Scenarios**:

1. **Given** 系統正常運作，**When** 使用者轉換指定時間 `2026-07-03 14:00`，從 `Asia/Taipei` 轉換至 `Europe/Paris`，**Then** 系統應正確考量巴黎與台北的時差，輸出對應的巴黎日期與時間。
2. **Given** 使用者需要對照多個時區，**When** 使用者指定了超過 3 個目標時區，**Then** 系統應以整齊的表格（Table）形式輸出，橫向或縱向顯示各時區的日期、時間與時差。

---

### User Story 3 - 尋找共同合適的會議時間 (Priority: P2)

使用者輸入多個與會者的地點與會議時長，系統自動推薦一個或多個在所有地點皆屬於「合理時間」的開會時段。

**Why this priority**: 解決跨時區會議排程時，需反覆計算與手動協調以避免成員在半夜開會的痛點。

**Independent Test**: 執行 `tz-cli schedule --locations "Taipei, London, New York" --date "2026-07-03" --duration "1h"`，系統應列出所有與會者皆處於合適時間的建議時段。

**Acceptance Scenarios**:

1. **Given** 使用者輸入三個分佈於亞洲、歐洲、美洲的地點，**When** 系統計算重疊時間，**Then** 系統應根據各時區的「合理開會時間範疇」，推薦重疊的綠色推薦時段。
2. **Given** 系統計算後發現沒有任何重疊的完美時間，**When** 尋找會議時間，**Then** 系統應依據「對所有人影響最小」的原則（如最少人需在深夜或清晨開會），輸出次優的建議時段。

---

### Edge Cases

- **地點名稱歧義**: 當使用者輸入 "Springfield" 時，系統應列出多個匹配的候選地點（如 Springfield, IL 或 Springfield, MA）供使用者選擇，或提示輸入更具體的格式。
- **無效輸入**: 使用者輸入不存在的郵遞區號或無法辨識的地點名稱時，系統應回傳明確的錯誤提示，而非程式崩潰，並列出相似的建議地點。
- **跨越日期變更線**: 會議推薦時間可能導致部分成員的日期與其他人不同（例如台北的 7/3 早上是紐約的 7/2 晚上），系統在輸出時必須清晰標註日期的差異（如 `+1 Day` 或 `-1 Day`）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須支援依據時區標識符（IANA 時區，如 `Asia/Taipei`）、通用城市/地點名稱（如 `New York`、`London`）以及美國 5 碼郵遞區號（如 `90210`）來解析並定位目標時區。
- **FR-002**: 系統必須能讀取指定時區的當前系統時間，並輸出格式化後的日期、時間、時區縮寫與相對 UTC 的偏移量（Offset）。
- **FR-003**: 系統必須支援將指定的日期時間（預設為使用者本機時間），轉換為多個指定目標時區或地點的對應時間，並以清晰的對照格式輸出。
- **FR-004**: 系統必須提供跨時區會議時間推薦演算法，依據使用者傳入的多個地點、日期與會議時長，自動計算並推薦重疊的「合理開會時間」。 [NEEDS CLARIFICATION: 會議合理開會時間的定義與自訂性]
- **FR-005**: 系統必須提供地點解析與查詢的本地快取機制，以減少對外部 API 的重複請求並支援部分離線功能。 [NEEDS CLARIFICATION: 地點與郵遞區號解析的資料來源]
- **FR-006**: 系統必須提供跨時區會議結果的匯出功能，方便使用者將排程好的時間快速帶入日常日程管理工具。 [NEEDS CLARIFICATION: 跨時區會議排程的整合輸出格式]

### Non-Functional & Quality Requirements

*根據專案憲法，本命令列工具必須符合以下品質關卡：*

- **NFR-001 (UX - CLI 一致性)**: 系統的命令列介面必須提供標準的 `--help` 幫助指南，清晰列出所有子命令與參數。所有錯誤訊息必須輸出至標準錯誤流（stderr），且成功執行時的結束代碼（Exit Code）必須為 `0`，參數錯誤為 `1`，解析或網路錯誤為 `2`。
- **NFR-002 (PERF - 效能要求)**: 本地時區查詢與時間換算必須在 50ms 內完成；若涉及外部地理編碼解析（如郵遞區號或模糊地點查詢），網路請求與解析總時間必須在 1 秒內完成，且重複查詢時應利用本地快取在 10ms 內響應。
- **NFR-003 (TEST - 測試標準)**: 核心時區轉換邏輯、美國郵遞區號映射邏輯，以及會議時間推薦演算法，必須具備至少 80% 的單元測試與整合測試覆蓋率。

### Key Entities

- **Location (地點)**: 代表一個地理位置。包含屬性：輸入名稱（如郵遞區號或城市名）、解析後的標準名稱、IANA 時區名稱（如 `America/New_York`）、經緯度（選填，用於輔助解析）。
- **TimeInstant (時間點)**: 代表一個特定的時間。包含屬性：時間戳記（UTC）、當地日期、當地時間、相對 UTC 偏移量。
- **MeetingProposal (會議提案)**: 代表一個建議的會議時段。包含屬性：會議開始時間（UTC）、會議時長、與會地點列表，以及各與會地點在該時段的本地時間與「合適度分數」。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001 (時區精確度)**: 所有的時區與夏令時間轉換必須 100% 精確，且完全符合當前 IANA 時區資料庫規範。
- **SC-002 (操作效率)**: 使用者從啟動工具到完成 3 個不同時區的共同會議時間查詢，所需的命令輸入與結果呈現總時間必須在 5 秒內完成。
- **SC-003 (模糊匹配成功率)**: 對於常見地點名稱的部分拼寫錯誤（如 "taipe"、"newyork"），系統的模糊搜尋正確匹配率必須大於 90%。

## Assumptions

- 系統在首次查詢新地點或郵遞區號時需要網路連線以進行地理編碼解析，後續則可透過快取離線執行。
- 美國郵遞區號與其對應時區的映射資料可以透過輕量級的嵌入式資料庫或外部 API 取得。
- 使用者本機系統時間是準確的，且做為計算相對時間的基準。
